# backend/CLAUDE.md

Guidance for the **backend** package. The backend connects to a Bilibili live room, converts its raw events into the typed SSE stream defined by **`docs/SCHEMA.md`**, and serves a selected frontend folder. Its only binding to any frontend is that SSE contract — see the top-level `CLAUDE.md` for the two-package / one-contract framing, the version axes, and the repo layout; this file covers the backend's internals.

Everything here lives under `backend/`. Launch with `python3 run.py` (the root shim that puts `backend/` on `sys.path` and calls `backend/main.py:main`) — or `python3 backend/main.py` directly.

## Modules

Eight single-responsibility modules in a strictly acyclic import graph, leaf → entry. They import each other flat (`from util import …`), which works because `backend/` is on `sys.path` (via `run.py`, or because it *is* the script dir when you run `backend/main.py`). Each module's top docstring states its responsibility; the shape:

- **`util.py`** — dependency-free leaf. Pure helpers that import nothing from this project so anyone may import them: `hhmm()`, `exception_summary()`, and the defensive parse helpers (`as_dict`, `parse_int`, `parse_float`) used while reading Bilibili's fragile nested payloads.
- **`schema.py`** — the SSE contract in code (the code side of `docs/SCHEMA.md`): the `EventType` enum, the `sender` shape (`sender()` / `system_sender()`), the `GuardTier` table + helpers mapping Bilibili's raw guard levels to the schema's `1/2/3` (and back to the Chinese tier name), and the `INIT_EVENT_ID` / `CENTS_PER_YUAN` / `SYSTEM_SENDER_*` constants. A near-leaf (stdlib only) shared by `bilibili.py` (the producer) and `stats.py` (a consumer) so the wire field names and the tier mapping live in exactly one place. **The field names here are the wire contract — keep them in lockstep with `docs/SCHEMA.md` and `API_VERSION`.**
- **`initialization.py`** — everything that must be right *before* anything is served: `BASE_DIR` and friends; `AppConfig` (the typed runtime-config container, **no field defaults**) and `ConfigLoader` (its only legitimate producer — reads `config.toml`, fails fast with `ConfigError` on any missing/malformed key); and `VersionGuard`, the two startup self-checks. The version/codename strings live in `main.py` and are **passed into** `VersionGuard` as arguments, so this module never imports `main.py` back.
- **`server.py`** — the whole browser-facing HTTP/SSE surface: `EventHub` (the in-process fan-out every event flows through — holds the single `init` event, a bounded history `deque` replayed to each new subscriber, and per-subscriber drop-oldest queues; assigns the monotonic `id` on `publish`) plus `create_flask_app()` (serves the selected frontend folder at the web root, `/stream` SSE, `/health`). `_sse()` is the wire encoding, private to this module.
- **`credentials.py`** — `CredentialManager`, the whole Bilibili credential lifecycle behind one public `obtain_credential()`: QR-code login (delegated to `bilibili_api.login_v2`), JSON persistence with an `obtained_at` stamp, and the freshness policy keyed on credential age — **< 24h** load as-is; **24h–7d** `check_refresh()`/`refresh()` then re-stamp; **≥ 7d** (or missing/unreadable) re-login by QR. Thresholds are `credential_load_max_age_seconds` / `credential_refresh_max_age_seconds`.
- **`stats.py`** — `StatsTracker`, thread-safe per-uid accumulation of gift/SC yuan and guard months; renders the on-screen report and writes it on stream end and on Ctrl+C. Knows nothing about where events come from — the adapter feeds it via `add()`. Its guard categories are keyed by tier name derived from `schema.GUARD_TIERS`, so there is no second tier list to keep in sync; `add()` ignores any unknown category.
- **`bilibili.py`** — the only place that knows Bilibili's raw formats. `BilibiliEventAdapter._convert` maps `DANMU_MSG`→`danmaku`, `SEND_GIFT`→`gift`, `SUPER_CHAT_MESSAGE`→`superchat`, `GUARD_BUY`→`guard`; the first three share one unified user parser (`_build_sender` over the UserInfo object), `GUARD_BUY` has no UserInfo and is built by hand. Also the room-info fetch/parse for the `init` event (`fetch_room_info` / `room_info_from_api_response` / `empty_room_info`) and the backend's own `system_message()` (reconnect notices, stream-end report, forwarded errors emitted as schema events). Parsing is defensive throughout — the field-mapping subtleties (guard-level inversion, now centralized in `schema.bili_guard_level_to_schema`; money units via `MILLIYUAN_PER_YUAN` / `schema.CENTS_PER_YUAN`; the SC dwell passthrough; missing-UserInfo) are documented at their call sites; cross-reference `docs/RAW_DATA.md`'s § notation. The output dicts are assembled here but their field names and `EventType`/`sender` shape come from `schema.py`.
- **`main.py`** — entry / orchestrator. Owns the version constants and `DanmakuHimeApp`, whose `run()` flow is: start Flask in a daemon thread → publish `init` (via `_build_init`) → obtain a `Credential` → `live.LiveDanmaku(...).on("ALL")` → `adapter.handle_event`, reconnecting forever. `main()` runs `VersionGuard.check_version(...)` (needs no config), loads config, runs `VersionGuard.check_frontend(...)`, then runs the app. Also `setup_logging()`: one process-wide `HH:MM:SS LEVEL message` format on the root logger, fanned to console (INFO+) and `log_file` (DEBUG+, appended); werkzeug muted to ERROR and `bilibili_api`'s `LiveDanmaku_*` logger rerouted via `_tame_lib_logger`. Use `log = logging.getLogger("danmakuhime")` — `info` for status, `warning` for recoverable oddities, `error(..., exc_info=True)` for failures (never `print` + `traceback.print_exc()`). Only interactive UI stays on raw `print()`: the QR block, the `\r` scan-poll spinner, and the stats report.

## Config

`config.toml` (alongside `main.py`, in `backend/`) is the backend runtime config — **no built-in defaults, env vars, or CLI flags**; `ConfigLoader.load()` fails fast rather than falling back. Basic keys are top-level, advanced ones under `[advanced]` (the loader checks both). **When adding a backend field, update `config.toml`, the `AppConfig` dataclass, and `_TOML_SCALAR_FIELDS` / `_TOML_PATH_FIELDS` together.** Fixed unit-conversion facts (`MILLIYUAN_PER_YUAN`, `schema.CENTS_PER_YUAN`) are code constants, not config; the backend sends no display durations except SuperChat's `dwell_seconds`, which is Bilibili's own authoritative value passed through (the only backend-chosen dwell is on its own system notices — `reconnect_notice_dwell_seconds` / `stream_end_report_dwell_seconds` / `debug_error_dwell_seconds`). Path fields resolve against `BASE_DIR` (= `backend/`); `frontend` is the directory selecting the served + integrity-checked frontend, and uniquely points *up and over* (`../frontends/<name>`) to the sibling `frontends/` folder. The frontend masthead (preprint title/category/authors) is *not* backend config — it lives in `frontends/preprint/config.json`. To forward backend errors to the UI, set `debug_forward_errors = true`.

## Build & integrity

After editing any backend `.py` you **must** rebuild, or the backend refuses to start:

```bash
python3 backend/build_backend.py    # regenerates backend.json (+ a backend-only package zip → dist/)
```

**`backend/build_backend.py` → `backend.json`** reads the version constants out of `main.py` by regex (so it stays dependency-free and runnable before the manifest exists) and hashes every module in `INTEGRITY_FILES`. `VersionGuard.check_version()` runs first in `main()` and refuses to start unless the versions and every module's sha256 match. The script is **backend-only** — it knows nothing about frontends; combining the backend with frontends into a bundle is the root `build.py`'s job.

The check is a **guard, not tamper-proofing** (`backend.json` is itself unprotected): it catches editing a module, bumping a version, or shipping a stale package without rebuilding. The hashing must stay **byte-identical** to `VersionGuard` — `sha256` over raw bytes (`read_bytes()`), no text decode or newline normalization; `file_sha256` here mirrors `initialization.py`'s `_file_sha256`. And **`INTEGRITY_FILES` in `initialization.py` must match `build_backend.py`'s exactly** — they are two copies on purpose (the backend ships independently of the frontend), so edit them together. (The frontend half of this same invariant lives in `frontends/CLAUDE.md`.)

Dependencies are in the root `requirements.txt` (lower-bound pins): `bilibili-api-python` (`import bilibili_api`, incl. `login_v2`), `Flask`, `Flask-Cors`, `pathspec`. Install with `python3 -m pip install -r requirements.txt`. After the first successful login a `credential.json` is written next to `main.py` (in `backend/`) and reused (see `credentials.py`).
